Getting Connected

The 3 commands between you and any node's BMC

There are only 3 commands between "I need to check a node" and having BMC data on your screen. Here's each one, what it does, and what you should see.

Step 1 — Authenticate

# Log in to Teleport (opens SSO in browser)
tsh login --proxy=teleport.na.int.coreweave.com teleport
This pops open your browser for SSO. Once you authenticate, tsh stores a short-lived certificate. The cert lasts ~12 hours — you only do this once per shift.
💡 Check if you're already authenticated: tsh status. Shows your cert expiry. "Not logged in" → run the login command above.

Step 2 — Land on the Jump Host

# SSH to the jump host at your site (EVI01)
tsh ssh acc@dh1-metal-jump01-us-central-07a
Now you're inside the DC network. The prompt changes — you're on the jump host. Every internal IP (including BMC IPs) is now reachable. The acc user has sudo access.
📍 Other sites have different jump hosts. Pattern: metal-jump01-{region}. Find yours: tsh ls | grep metal-jump
EVI01 dh1-metal-jump01-us-central-07a
US-EAST metal-jump01-us-east-04a
RNO2 metal-jump01-rno2
RDU1 metal-jump01-rdu1

Step 3 — Talk to the BMC

# Option A: wreakit (happy path — handles creds for you)
wreakit bmc <node-name> "sel list"

# Option B: raw ipmitool (when wreakit fails or you need full control)
ipmitool -I lanplus -H $BMC_IP -U $BMC_USER -P "$BMC_PASS" sel list
wreakit is your friend — it wraps ipmitool and handles credentials automatically. Use it first. If it fails (agent down, node not in inventory), fall back to raw ipmitool. Get the BMC IP from NetBox.
🔗 Where do BMC IPs come from? NetBox is CW's source of truth for device info. Look up any node to find its BMC/IPMI interface IP.

DCT Command Reference

Every ipmitool command you'll actually use on shift

All of these run from the jump host. Replace $BMC with ipmitool -I lanplus -H $IP -U $USER -P "$PASS". Or use wreakit bmc <node> "command".
🔍 SEL List (Event Log)
$BMC sel list $BMC sel list last 20 $BMC sel clear
First thing on any sick node. Shows hardware events.
⚡ Power Control
$BMC chassis power status $BMC chassis power cycle $BMC chassis power off $BMC chassis power on
Node unresponsive, needs hard reboot.
🌡️ Sensor Readings
$BMC sdr list $BMC sdr type Temperature $BMC sdr type Fan $BMC sdr type Voltage
Thermal alerts, fan failures, power instability.
🔄 BMC / MC Reset
$BMC mc reset cold $BMC mc reset warm $BMC mc info
BMC itself hung or returning garbage.
📋 FRU Info (Part Numbers)
$BMC fru list $BMC fru print 0
RMA ticket — need serial numbers, model, manufacturer.
🖥️ SOL Console (Serial)
$BMC sol activate $BMC sol deactivate
Boot output, BIOS errors, kernel panics live. Exit: ~.
💡 Chassis LED / Identify
$BMC chassis identify 30 $BMC chassis identify force
Physical locate — blinks chassis LED.
🔌 Boot Device Override
$BMC chassis bootdev pxe $BMC chassis bootdev disk $BMC chassis bootdev bios
Force PXE reinstall, boot to BIOS, restore disk boot.
🔗 wreakit wraps all of these. wreakit bmc node123 "sel list" = same as raw ipmitool, but handles creds automatically.

SEL Event Decoder

What the System Event Log is telling you

The SEL is the black box recorder for server hardware. Every thermal spike, power fault, and memory error gets logged with a timestamp. This is the first place you look on a sick node.
Processor | IERR
Critical
CPU internal error — hardware fault
Action: Cold reset. If repeats → RMA.
Power Supply | Failure
Critical
PSU dead — node on single supply or down
Action: Physical swap. Non-redundant → urgent.
Memory | Uncorrectable ECC
Critical
RAM error couldn't be corrected — data corruption
Action: DIMM replacement. Pull slot info for RMA.
Temperature | Upper Critical
High
Emergency thermal threshold hit
Action: Check fans + ambient. Physical inspect if persistent.
OS Stop / Shutdown
High
Kernel panic, OOM, or watchdog timeout
Action: SOL for OS logs. Check dmesg. Note if recurring.
Memory | Correctable ECC
Medium
RAM bit flip corrected — benign if rare
Action: Log count. Rate increases → schedule DIMM swap.
Fan | Lower Critical
Medium
Fan RPM below safe threshold
Action: Identify slot. Urgent if temps follow.
System Boot | Initiated
Info
Node powered on or rebooted
Action: Confirm expected. Unexpected → check prior events.
Chassis | Intrusion
Info
Physical chassis opened
Action: Match against work orders.
Drive Slot | Presence
Info
Drive inserted or removed
Action: Correlate with maintenance.
Critical — act now
High — same session
Medium — monitor
Info — log & verify
Reading SEL is pattern recognition. "Fan Lower Critical → Temperature Upper Critical → OS Stop" = fan died → node overheated → kernel panicked. The events tell the story in order.

tsh Command Cheat Sheet

Everything the Teleport CLI can do

Teleport isn't just SSH. It handles SSH, Kubernetes, web apps, databases, and cloud APIs — all with short-lived certs and full audit logging. tsh is your entry point.
🔐 Auth & Session
tsh login --proxy=teleport.na.int.coreweave.com teleport tsh logout tsh status tsh env tsh config
Start of shift: login. End: logout. Check: status.
🖥️ SSH & File Transfer
tsh ssh acc@<hostname> tsh ssh acc@<host> <command> tsh ls tsh ls | grep metal-jump tsh scp file.sh acc@host:/tmp/
Daily use. tsh ls shows what your roles let you reach.
☸️ Kubernetes
tsh kube ls tsh kube login us-central-07a-mgmt tsh kubectl get nodes tsh kubectl get pods -A
Your cluster: us-central-07a-mgmt. K8s enabled.
🌐 Web Applications
tsh apps ls tsh apps login grafana tsh proxy app grafana -p 8080
Grafana, internal tools — anything behind Teleport.
🗄️ Databases
tsh db ls tsh db login --db-user=rpatino mydb tsh db connect --db-user=admin mydb tsh proxy db mydb -p 5432
PostgreSQL, MySQL, MongoDB, Redis. No passwords needed.
🎫 Access Requests
tsh request create --roles=<role> \ --reason="BMC access for CW-1234" tsh request ls tsh request show <id>
Auto-triggered on "access denied". Submit reason, await approval.
📼 Session Recordings
tsh recordings ls tsh play <session-id> tsh play --skip-idle-time <id>
Review support sessions. All sessions recorded.
🔧 Diagnostics & Proxy
tsh bench --duration=10s acc@host tsh proxy ssh acc@host tsh proxy app <app> -p <port> tsh proxy kube <cluster> -p <port>
Benchmarking, local proxying, debug connectivity.
